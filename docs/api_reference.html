Device registration

POST /vitalsync/v1/devices/register Register a new device ▼
Registration is closed by default. Use a one-time iphone pairing token, or use admin token/open registration for maintenance and local testing. Returns device_id + long-lived refresh_token + initial access_token.

Request body

FieldTypeDescription
schemastringrequired"vitalsync.device_registration.v1"
pairing_tokenstringoptionalRequired for normal closed-registration iPhone pairing
device_labelstringrequiredHuman-readable label, e.g. "My iPhone"
platformstringrequired"iOS"
app_versionstringrequiredSemver string, e.g. "0.1.0"
{
  "schema": "vitalsync.device_registration.v1",
  "pairing_token": "<PAIRING_TOKEN>",
  "device_label": "My iPhone",
  "platform": "iOS",
  "app_version": "0.1.0"
}

Response 200

{
  "device_id":     "dev_01J...",
  "refresh_token": "vitalsync_refresh_...",
  "access_token":  "vitalsync_access_...",
  "scopes":        ["write:healthkit"],
  "expires_at":    "2026-06-26T12:00:00Z"
}
POST /vitalsync/v1/devices/revoke Revoke a device (app-initiated) ▼
Requires valid access token for the device being revoked. Server sets revoked_at on device and invalidates all tokens.

Request body

{ "device_id": "dev_01J..." }

Response 200

{ "revoked": true, "device_id": "dev_01J..." }

Client registration

POST /vitalsync/v1/clients/register Register a read-only API client ▼
Use an ingest pairing token with read:healthkit. Clients are API consumers, not HealthKit producer devices.

Request body

{
  "schema":        "vitalsync.client_registration.v1",
  "pairing_token": "vitalsync_pair_...",
  "client_type":   "ingest",
  "client_label":  "ingest on receiver.example.com"
}

Response 200

{
  "client_id":     "client_01J...",
  "client_type":   "ingest",
  "refresh_token": "vitalsync_refresh_...",
  "access_token":  "vitalsync_access_...",
  "scopes":        ["read:healthkit"],
  "expires_at":    "2026-06-26T12:00:00Z"
}

Token endpoints

POST /vitalsync/v1/tokens/session Get short-lived WebTransport session token ▼
Session token lifetime: 5 minutes. Used as URL query param on WebTransport connection (browser WT API has limited custom header support).

Request body

{
  "device_id": "<DEVICE_ID>",
  "capability": "webtransport_upload"
}

Response 200

{
  "session_token": "<SESSION_TOKEN>",
  "expires_at":    "2026-06-26T10:17:00Z",
  "transport_url": "https://receiver.example.com/vitalsync/v1/transport?token=<SESSION_TOKEN>"
}
POST /vitalsync/v1/tokens/refresh Exchange refresh token for new access token ▼

Request body

{
  "refresh_token": "<REFRESH_TOKEN>",
  "device_id":     "<DEVICE_ID>"
}

Response 200

{
  "access_token": "<ACCESS_TOKEN>",
  "expires_at":   "2026-06-26T11:00:00Z"
}

WebTransport upload

Endpoint: https://receiver.example.com/vitalsync/v1/transport?token=<session-token>
Uses reliable bidirectional streams. Health payloads must not use datagrams.

Control stream — hello

// App → Server (first message on control stream)
{
  "type":        "hello",
  "schema":      "vitalsync.control.v1",
  "device_id":   "<DEVICE_ID>",
  "app_version": "0.1.0",
  "supports":    ["health.batch.v1", "tombstone.v1"]
}

// Server → App
{
  "type":            "hello_ack",
  "server_time":    "2026-06-26T10:12:30Z",
  "max_batch_bytes": 1048576
}

Batch upload stream

// App → Server
{
  "type":       "batch",
  "schema":     "vitalsync.batch.v1",
  "batch_id":   "batch_01J...",
  "device_id":  "dev_01J...",
  "created_at": "2026-06-26T10:12:30+09:00",
  "records":    [ /* record objects */ ],
  "deleted":    [ /* tombstone objects */ ]
}

// Server → App
{
  "type":      "batch_ack",
  "batch_id":  "batch_01J...",
  "accepted":  120,
  "deleted":   3,
  "duplicate": false
}

Batch upload (HTTPS fallback)

POST /vitalsync/v1/batches Upload batch when WebTransport unavailable ▼
Exact same batch JSON schema as WebTransport. Idempotency-Key header is required and must equal batch_id. Auth requires a Bearer access token with write:healthkit.

Headers

HeaderValue
AuthorizationBearer vitalsync_access_...
Content-Typeapplication/json
Idempotency-Key<batch_id> — required

Request body

{
  "schema":     "vitalsync.batch.v1",
  "batch_id":   "batch_01J...",
  "device_id":  "dev_01J...",
  "created_at": "2026-06-26T10:12:30+09:00",
  "timezone":   "Asia/Tokyo",
  "sequence":   42,
  "records":    [],
  "deleted":    []
}

Response 200

{
  "batch_id":  "batch_01J...",
  "accepted":  120,
  "deleted":   3,
  "duplicate": false
}

Conflict (same batch_id, different payload)

{
  "type":    "error",
  "code":    "batch_id_conflict",
  "message": "batch_id already exists with different payload"
}

Consumer tokens

POST /vitalsync/v1/consumer-tokens Issue a scoped read token for downstream consumers ▼
Requires admin token. Consumer tokens are read-only and scoped to specific data types.

Request body

{
  "scope":              ["read:sleep", "read:activity", "read:vitals"],
  "expires_in_seconds": 3600
}

Response 200

{
  "access_token": "vitalsync_consumer_...",
  "expires_at":   "2026-06-26T11:12:30Z"
}

Record fetch

GET /vitalsync/v1/records Fetch normalized records by type and time range ▼
Requires read:healthkit or a legacy consumer token with the appropriate granular read scope.

Query parameters

ParamTypeDescription
sample_typestringrequirede.g. sleep_analysis, step_count
startdateoptionalISO 8601 date or datetime
enddateoptionalISO 8601 date or datetime
cursorstringoptionalPagination cursor from previous response
limitintegeroptionalMax records per page (default 500)

Example request

GET /vitalsync/v1/records?sample_type=sleep_analysis&start=2026-06-01&end=2026-06-26
Authorization: Bearer vitalsync_consumer_...

Response 200

{
  "schema":      "vitalsync.records.v1",
  "records":     [ /* record objects */ ],
  "next_cursor": null  // or opaque cursor string
}

Batch fetch

GET /vitalsync/v1/batches Fetch raw immutable batches since a timestamp ▼
Returns raw batch envelopes as stored. Useful for reprocessing or debugging. Requires read:healthkit.

Query parameters

ParamTypeDescription
sincedatetimeoptionalISO 8601 datetime — fetch batches received after this time
device_idstringoptionalFilter to a specific device
cursorstringoptionalPagination cursor

Example

GET /vitalsync/v1/batches?since=2026-06-01T00:00:00Z
Authorization: Bearer vitalsync_consumer_...

Admin endpoints

POST /vitalsync/v1/admin/pairing-tokens Issue a one-time registration token ▼
Requires admin:tokens. Stores only token hash. Valid client types are iphone with write:healthkit and ingest with read:healthkit.

Request body

{
  "schema":      "vitalsync.pairing_token_request.v1",
  "client_type": "iphone",
  "scopes":      ["write:healthkit"],
  "ttl_seconds": 600
}

Response 200

{
  "schema":           "vitalsync.pairing_token.v1",
  "pairing_token":    "<PAIRING_TOKEN>",
  "client_type":      "iphone",
  "scopes":           ["write:healthkit"],
  "expires_at":       "2026-06-27T15:30:00Z",
  "registration_url": "vitalsync://register?base_url=..."
}
POST /vitalsync/v1/devices/{device_id}/revoke Admin-revoke a device ▼
Requires admin:devices scope. Sets revoked_at, invalidates all tokens for device.
POST /vitalsync/v1/devices/{device_id}/purge Delete all data for a device ▼
Requires admin:devices scope. Deletes device registration, tokens, healthkit_batches, and healthkit_records rows for device. Irreversible.
POST /vitalsync/v1/purge Purge by sample type ▼

Query parameter

POST /vitalsync/v1/purge?sample_type=sleep_analysis
Requires admin:devices scope. Deletes all records of the given sample type across all devices.

  "code":      "unsupported_schema",

Database schema

-- Raw immutable batches
healthkit_batches (
  batch_id        text primary key,
  device_id       text not null,
  created_at      timestamptz not null,
  received_at     timestamptz not null,
  schema          text not null,
  payload_json    jsonb not null,
  payload_sha256  text not null
)

-- Normalized record index
healthkit_records (
  source          text not null,
  source_id       text not null,
  device_id       text not null,
  sample_type     text not null,
  source_bundle_id text,
  source_name     text,
  start_time      timestamptz,
  end_time        timestamptz,
  timezone        text,
  value_json      jsonb not null,
  metadata_json   jsonb not null,
  batch_id        text not null,
  deleted         boolean not null default false,
  updated_at      timestamptz not null,
  primary key (source, source_id)
)

-- Devices
healthkit_devices (
  device_id    text primary key,
  device_label text,
  app_version  text,
  platform     text,
  created_at   timestamptz,
  revoked_at   timestamptz
)

-- Tokens (hash only — never store plaintext)
healthkit_tokens (
  token_hash  text primary key,
  device_id   text,
  token_type  text not null,  -- 'refresh' | 'access' | 'session' | 'consumer'
  scopes_json json not null,
  expires_at  timestamptz,
  revoked_at  timestamptz
)

-- One-time pairing tokens (hash only — never store plaintext)
pairing_tokens (
  token_hash  text primary key,
  client_type text not null,
  scopes_json json not null,
  created_at  timestamptz not null,
  expires_at  timestamptz not null,
  consumed_at timestamptz
)

-- API clients
clients (
  client_id    text primary key,
  client_type  text not null,
  client_label text,
  created_at   timestamptz not null,
  revoked_at   timestamptz
)

-- Client tokens (hash only — never store plaintext)
client_tokens (
  token_hash  text primary key,
  client_id   text not null,
  token_type  text not null,
  scopes_json json not null,
  expires_at  timestamptz,
  created_at  timestamptz not null,
  revoked_at  timestamptz
)
